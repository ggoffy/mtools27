# mtools Codex Tasks

Context: `mtools` is currently acting as three things at once: a shared runtime dependency, a testing ground for candidate support classes, and a normal XOOPS module with its own UI and preload behavior. The concept has value, and `quotes` proves reuse is possible, but the boundary must be tightened before more modules depend on it.

Implementation goal: make `mtools` a stable shared-module layer for vetted support classes, keep experimental work clearly separated, and define a clean graduation path into XMF once an API has proven its value.

1. [x] Define the architecture contract for `mtools` in `README.md` and a new `docs/architecture.md`: state explicitly which classes are public shared API, which are experimental, which are module-local, and what “ready to move to XMF” means.
2. [x] Split the namespace surface into at least two tiers: keep stable reusable code under `XoopsModules\Mtools\Common\...`, move incubating code into `XoopsModules\Mtools\Experimental\...` or `Lab\...`, and stop exposing unfinished helpers as if they were part of the public contract.
3. [x] Create one dedicated bootstrap entrypoint such as `modules/mtools/bootstrap.php` that does only autoload registration and dependency/version checks; consumer modules must require this bootstrap instead of hard-coding `preloads/autoloader.php`.
4. [x] Remove hidden global side effects from `mtools` preload behavior before other modules depend on it: `preloads/core.php` must not inject jQuery/Bootstrap, write to the database, or alter every page request just because `mtools` is installed.
5. [x] Keep autoload registration separate from UI/theme/bootstrap features: if theme helpers are still needed, move them into an explicit opt-in service instead of `MtoolsCorePreload::eventCoreFooterStart()`.
6. [x] Formalize dependency handling for consumer modules such as `quotes`: add an install/update prerequisite check for `mtools` being installed, active, and at a minimum supported version, instead of relying on README text plus `class_exists()` checks in admin bootstrap.
7. [x] Introduce a public API version constant or helper in `mtools` so consumer modules can assert compatibility without guessing from the module version string.
8. [x] Document and enforce the autoloader contract: `mtools` owns registration of `XoopsModules\Mtools\...`, consumer modules own their own namespaces, and runtime loading must not depend on preload order accidents. Add a defensive autoloader registration path for CLI/tests where XOOPS preloads may not fire.
9. [x] Decide the fate of `mtools_header.php`: either delete it as an unused dead-end or turn it into the documented public bootstrap entry point with a clear contract and a version constant.
10. [x] Make every shared class library-safe: class files in `class/Common/` must stop handling requests or redirects at file scope. `class/Common/FileChecker.php` and `class/Common/DirectoryChecker.php` currently behave like controllers and must be split into pure classes plus separate admin endpoints.
11. [x] Remove browser-posted filesystem actions from the shared layer. `FileChecker`, `DirectoryChecker`, and related filesystem helpers must never accept arbitrary posted paths or modes; replace them with allowlisted keys, `realpath()` containment, and least-privilege permissions.
12. [ ] Audit `class/Common/FilesManagement.php` and `class/Common/SysUtility.php` for promotion readiness: remove `queryF()`, raw SQL string assembly, direct output, direct redirects, broad globals, and other module-era behavior that does not belong in a shared runtime library. Initial pass removed `queryF()` usage, hardened filesystem paths, added fetch guards, and fixed clone/table helper SQL; broad globals and UI-era helpers still need a deeper API split.
13. [x] Rework `class/Common/Breadcrumb.php` so it does not infer the rendering template from the `mtools` dirname. A shared UI helper must accept the consuming module context or a renderer/template name explicitly, otherwise it remains tied to `mtools`.
14. [x] Keep the good shared patterns and harden them: `Configurator` is a solid candidate for reuse, but it should be documented as a stable contract and backed by typed DTO-like config loading rather than loose mixed properties.
15. [x] Move optional remote/version-check logic out of the foundational trait layer. `class/Common/VersionChecks.php` should keep local XOOPS/PHP requirement checks, while GitHub release polling should become an optional updater service, not part of every shared utility trait.
16. [x] Bring `mtools` itself to the current XOOPS baseline: rewrite `composer.json` with real package metadata, `php` constraint `>=8.2 <8.6`, explicit autoload rules, proper `require-dev`, and only the runtime dependencies that truly belong in production.
17. [x] Decide whether `pmjones/auto-route` is a real runtime requirement or only an experimental/dev dependency, and move it out of production runtime if it is not part of the stable shared API.
18. [ ] Separate “shared runtime” from “mtools app” code. Module-specific UI, theme helpers, bootstrap assets, and legacy convenience methods in `class/Utility.php` should remain internal to `mtools` or move to an experimental area, not be treated as the reusable base for other modules.
19. [x] Clean the distribution boundary: do not ship `vendor/`, `node_modules/`, generated vendor tests, `composer.lock`, or JS build artifacts in the installable module archive. Add proper `.gitattributes export-ignore` rules and keep release payloads focused on runtime files only.
20. [ ] Add contract tests for the shared API using `quotes` as the first consumer: verify bootstrap loading, install/update dependency failure modes, `Configurator`, `VersionChecks`, and any retained shared helpers without relying on mtools-specific UI side effects. Initial bootstrap/inheritance contract added in `tests/Contract/QuotesConsumerContractTest.php`; broaden remaining cases. Smoke checks now verify mtools bootstrap, quotes bootstrap, Utility inheritance, and clean unbootstrapped dependency failure.
21. [ ] Add a proper quality gate for the shared layer: one PHPUnit smoke test per promoted helper, CI on PHP 8.2 through 8.5, and at least one explicit failure-mode test for “consumer installed without compatible mtools”.
22. [x] Refactor `quotes` to use the new bootstrap and dependency contract. Replace direct `require_once XOOPS_ROOT_PATH . '/modules/mtools/preloads/autoloader.php'` usage and ad hoc admin checks with one reusable dependency helper.
23. [x] Publish a migration guide for other modules: how to consume `mtools`, which classes are approved for reuse, which ones are internal/experimental, and how to prepare for later XMF migration without another round of broad rewrites.
24. [x] Add explicit versioning and deprecation policy docs such as `docs/VERSIONING.md`: semver rules for shared helpers, `@deprecated` usage, changelog discipline, and compatibility expectations for consuming modules.
25. [x] Define graduation criteria for XMF candidates: a class should move from `mtools` only after it has at least one real consumer module, a stable constructor/API, no module-relative assumptions, no request-side effects, tests, and documented behavior.
26. [ ] After the API surface is clean and tested, introduce thin compatibility adapters or `class_alias` bridges so future moves from `XoopsModules\Mtools\Common\...` to `Xmf\...` can be done with deprecations instead of breaking every consumer module at once.
